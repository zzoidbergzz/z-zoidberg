# z.je Critical Evaluation — Full Report
## Date: 2026-05-10

## System Overview

**z.je** is a self-hosted threat intelligence knowledge platform built on FastAPI + PostgreSQL + Redis + ARQ workers. It ingests CTI feeds (RSS, NVD, Exploit-DB, KEV, MISP, Tor scrapes), extracts IOCs via regex/NER extractors, materializes them as entities in a knowledge graph, and enriches them via 16+ provider integrations (VirusTotal, Shodan, GreyNoise, AbuseIPDB, URLScan, CrowdStrike, MITRE ATT&CK, NVD, OTX, BGP HE, IPInfo, Recorded Future, MISP, OpenCTI). It exposes REST, GraphQL, STIX 2.1 export, TAXII 2.1, and MCP SSE endpoints, with a Jinja2 server-rendered UI.

**At rest:** 44,209 entities, 124,026 relationships, 396,024 corpus documents (349K CVEs, 46K Exploit-DB), 118,018 claims, 7,702 enrichment cache entries, 8 users, 7 watchlists.

## PERSONA 1: SOFTWARE DEVELOPER

### What's Working Well

1. **Architectural breadth is impressive.** Multi-tenant RLS, async SQLAlchemy, ARQ workers, BFS graph traversal, OpenTelemetry, BYOK key management — this isn't a weekend project.
2. **Provider surface area.** 16 enrichment providers with budget controls, TTL caching, and diff tracking. That's real operational tooling.
3. **Ingestion pipeline.** Feed polling, Tor scraping, corpus refresh, idempotent upserts, content dedup by SHA-256 — solid data engineering.
4. **API surface.** REST, GraphQL, STIX, TAXII, MCP. That's unusual completeness for a personal platform.
5. **Test suite.** 40+ test files covering auth, enrichment, extractors, graph, search, webhooks, watchlists.

### What's Weak / Missing

6. **Duplicate entities.** 20 duplicate `(kind, canonical_name)` pairs exist *despite* the unique constraint in migration 0026. The `on_conflict_do_nothing` in the worker is silent — when it fires, the entity already exists, but the worker still creates a Claim pointing at it, and then does a second `SELECT` to load it. The constraint clearly isn't enforced on all tenant partitions or was added after the dupes were created. The top offender (6 dupes each for `www.barnessolicitors.co.uk`, Rhysida onion domains) suggests the Tor scraper is inserting entities before the main ingest pipeline runs, creating races.

7. **Relationship cardinality is pathological.** 99,433 of 124,026 relationships (80%) are `investigation_context_for` — a single relationship type that's essentially a bag of context, not a semantic edge. This makes the graph mostly hairball; only 24,593 edges carry actual threat-intel semantics (uses, detects, references_technique, etc.). BFS on 124K edges with 80% junk is expensive for no analytical payoff.

8. **`observed_in_document` claims dominate (44,672 of 118,018).** These are low-value "I saw this entity in a doc" assertions. They're not actionable intelligence — they're bookkeeping. They inflate the claims table, slow search (every claim is JOIN'd in the search SQL), and add no analytical value.

9. **Search is slow.** The search SQL runs 4+ correlated subqueries per entity row (3 `SELECT ... FROM claims` subqueries in `_ENTITY_SQL`), then does claim search, corpus search, and ILIKE fallback. On 44K entities + 118K claims + 396K corpus docs, this is multi-hundred-millisecond territory. No query plan caching, no materialized views, no vector/semantic search wired up (embeddings code exists but isn't connected to the search endpoint).

10. **Entity model is too flat.** `external_refs` is a JSONB grab-bag. CVE severity, actor country, malware family — all stuffed into the same field with no schema enforcement. You can't query "all critical CVEs" or "all Chinese APTs" efficiently.

11. **No pagination on graph traversal.** The `build_graph` function loads all relationships for a tenant into memory when no entity_id is specified. With 124K relationships, this will OOM under load. Even focused BFS with `depth=3` can explode.

12. **Digest runner is a stub.** `run_pending_digests` creates a `DigestRun` row, sets `status="complete"`, `item_count=0`, and returns. It never builds content or dispatches. The entire digest subsystem (subscription, scheduler, runner) is scaffolding.

13. **Ask endpoint is a raw proxy.** It sends the question straight to Anthropic with zero RAG. No context injection from the knowledge base, no retrieval, no guardrails. The `/ask` endpoint is a thin wrapper around `curl api.anthropic.com`.

14. **MCP bridge connects to Darkmoon but doesn't pull results back.** The bridge fires off enrichment requests but the MCP server doesn't expose a "get results" flow that a remote MCP client can poll.

15. **CORS is `allow_origins=["*"]`.** On a multi-tenant platform with auth. This is a cross-origin nightmare if any tenant hosts malicious content.

16. **Path finding loads all relationships.** `find_path` does `SELECT * FROM relationships WHERE tenant_id = :tid` — that's 124K rows into a NetworkX graph in Python, every time. Should be a recursive CTE in Postgres.

17. **No rate limiting on the API.** The UI has session auth, but the API key path has no per-key rate limits. A compromised key can drain enrichment budgets.

18. **Embedding code is orphaned.** `app/embeddings/generator.py` and `search.py` exist but aren't called from any router or worker. The config has `EMBEDDING_PROVIDER`, `EMBEDDING_MODEL`, `EMBEDDING_DIM` — all wired but dead code.

## PERSONA 2: CYBER THREAT ANALYST

### What's Working Well

1. **IOC extraction pipeline.** 17 extractors (CVE, CPE, CWE, EUVD, domain, IP, MD5, SHA256, URL, technique, tactic, malware, actor, GitHub advisory, NVD CVE) — that's a solid extraction surface. Defanged-input handling is a nice touch.
2. **Tor scraping + AI enrichment.** Deterministic parsing (victims, wallets, hashes, binaries) plus LLM-based enrichment of onion pages — this is genuinely useful for ransomware tracking.
3. **Multi-provider enrichment with budget guards.** For a solo operator, the budget system (per-provider daily/minute caps, TTL-based caching) prevents API key exhaustion. Real operational discipline.
4. **Watchlists + IOC pingback.** The watchlist-→-pingback pipeline (new IOC matches → notifies watchers) is a solid alerting primitive.
5. **STIX 2.1 export.** Interoperable with MISP, OpenCTI, ThreatConnect. Not just a silo.

### What's Weak / Missing

6. **No confidence scoring framework.** All relationships default to `confidence=1.0`. There's no uncertainty modeling. When you have "APT28 uses Mimikatz" (from a MITRE attestation) and "8.8.8.8 resolves to dns.google" (from Shodan) both at confidence=1.0, the graph is lying. Analysts need graduated confidence (low/medium/high or 0-100) with source attribution.

7. **No temporal modeling.** Entities and relationships have `created_at` but no `valid_from`/`valid_until`. An IP that was malicious in 2023 may be clean now. A domain that resolved to a C2 yesterday may be sinkholed today. Without temporal scoping, the knowledge base degrades over time.

8. **No TTP-to-detection mapping at the entity level.** MITRE ATT&CK techniques are ingested as entities, and there are `detectable_by` / `detects` relationship types — but only 2,301 + 1,525 = 3,826 such edges vs. 1,400 attack_pattern entities. That's ~2.7 detections per technique on average, which is thin. Most techniques have zero detection coverage.

9. **Corpus documents are disconnected from the knowledge graph.** 396K corpus docs (CVE, Exploit-DB) are searchable via FTS but aren't linked to entities via relationships. CVE-2024-3094 (XZ Utils) should be a node in the graph connected to the `xz` tool entity, the `supply_chain` tactic, and the `Fedorov` actor. Right now it's a text blob in `corpus_documents` with no graph edges.

10. **No analyst workflow / case management.** Investigations exist as a data model, but there's no timeline view, no evidence board, no narrative builder. An analyst tracking an incident can't build a story — they can only add entities to a flat list.

11. **No feedback loop on enrichment quality.** Enrichment cache entries have `success` boolean but no quality rating. If Shodan returns stale data, or VirusTotal flags a false positive, there's no mechanism to down-rank or flag it. Bad data persists until TTL expiry.

12. **No threat model / adversary profile view.** There are 920 threat_actor entities, but no dedicated UI to see "all TTPs used by APT28" or "all victims of LockBit" as a structured view. The entity detail page shows raw JSON.

13. **Onion data is high-risk and under-controlled.** Tor scraping stores raw body text, screenshots, and artifact files. There's no data classification, no retention policy, no legal review workflow. If a .onion page contains CSAM or classified material, there's no automated detection or redaction.

14. **No kill chain / campaign timeline.** The relationship types include `uses`, `references_technique`, `co_occurs_with` — but there's no temporal ordering. You can't reconstruct "first they phished, then they dropped Mimikatz, then they exfiltrated." Campaigns need ordered phases, not just unordered edges.

15. **Sector/ISAC features are stub-level.** The sectors model exists, invite codes exist, but there's no actual sector-based data sharing, no cross-tenant visibility, no ISAC-specific dashboards. The entire multi-tenant RLS infrastructure is in service of single-tenant operation right now.

16. **No indicator lifecycle management.** IOCs appear, get enriched, and sit forever. There's no lifecycle states (active, expired, retired, false_positive). An analyst can't mark "8.8.8.8 is a known benign — never alert on this."

17. **CVE data is stale-reliant on NVD refresh.** The daily corpus refresh runs at 03:17 UTC, but NVD's API is rate-limited and frequently returns 403s. There's no fallback to OSV or CVE.org direct. The 349K CVE records may have gaps.

## 30 DETAILED PROPOSED CHANGES

### Data Quality & Integrity

**1. Deduplicate entities and enforce unique constraint at DB level**
- **What:** Add `UNIQUE (tenant_id, kind, canonical_name)` constraint directly in the DB if not already there. Write a one-time migration that merges duplicate entity rows (re-point all claims, relationships, enrichment_cache entries to the survivor, DELETE the losers). The current 20 duplicate groups (6-way splits for Rhysida domains) indicate the Tor scraper and main ingest are racing.
- **Why:** Duplicate entities break graph traversal (two nodes for the same domain = split edges), inflate entity counts, and confuse search (two results for the same thing). The `on_conflict_do_nothing` in the worker silently swallows the race instead of resolving it.
- **How:** Migration script: `SELECT kind, canonical_name, array_agg(id ORDER BY created_at) as ids FROM entities GROUP BY kind, canonical_name HAVING count(*) > 1`. For each group, keep the oldest `id` as survivor. `UPDATE claims SET entity_id = survivor WHERE entity_id = loser`. Same for relationships (from_entity_id, to_entity_id) and enrichment_cache. `DELETE FROM entities WHERE id = ANY(losers)`. Add `UNIQUE` constraint.

**2. Prune `investigation_context_for` to a separate lightweight table**
- **What:** Move the 99,433 `investigation_context_for` relationship rows into a new `investigation_context` table `(investigation_id, entity_id, added_at)` and delete them from `relationships`. This drops the relationships table from 124K to ~25K rows, all semantically meaningful.
- **Why:** 80% of the graph is context-grouping edges with no analytical value. They make BFS expensive, make the graph UI a hairball, and dilute relationship statistics. Investigation context is not a "relationship" in the CTI sense — it's a bag membership.
- **How:** `CREATE TABLE investigation_context AS SELECT ... FROM relationships WHERE kind = 'investigation_context_for'`. Drop those rows from relationships. Update the investigation UI to query `investigation_context` directly.

**3. Add indicator lifecycle state to entities**
- **What:** Add `lifecycle_state VARCHAR(20) DEFAULT 'active'` to entities with values: `active`, `expired`, `retired`, `false_positive`, `benign`. Add an API endpoint `PATCH /api/v1/entities/{id}/lifecycle` and a UI dropdown.
- **Why:** IOCs decay. An IP that was a C2 in 2023 may be a residential ISP now. Without lifecycle tracking, analysts get desensitized to stale alerts. This is table-stakes for any CTI platform.
- **How:** Migration adds the column. The lookup endpoint returns the state. Enrichment diff detection can auto-suggest `expired` when enrichment data changes significantly. Watchlist notifications skip `retired`/`false_positive` entities.

**4. Add `valid_from`/`valid_until` temporal columns to relationships**
- **What:** Add `valid_from TIMESTAMPTZ` and `valid_until TIMESTAMPTZ` to relationships. Default `valid_from = NOW()`, `valid_until = NULL` (open-ended). Add query parameter `?at=2024-06-01T00:00:00Z` to relationship endpoints that filters to relationships active at that time.
- **Why:** Threat intelligence is temporal. "APT28 uses Mimikatz" was true in 2020 but their toolset evolved. Without temporal scoping, the knowledge base is a timeless mush. STIX 2.1 relationships already have `start_time`/`stop_time` — this maps directly.
- **How:** Migration adds columns. The STIX builder already has the mapping infrastructure. The graph visualization highlights time-scoped edges differently (dashed for expired, solid for current).

**5. Graduated confidence scoring with source provenance**
- **What:** Replace the `confidence FLOAT DEFAULT 1.0` on relationships with a `confidence VARCHAR(10) DEFAULT 'medium'` enum (`low`, `medium`, `high`, `unknown`) plus a `source_type VARCHAR(50)` column (e.g., `mitre_attestation`, `enrichment_provider`, `analyst_judgement`, `co_occurrence`).
- **Why:** Confidence=1.0 for everything is meaningless. A Shodan port scan confirming "port 443 open" is high confidence. A co-occurrence inference that "this IP appeared near that domain" is low confidence. Analysts need to weight evidence.
- **How:** Migration. Update the search results API to include confidence. The graph visualization uses edge thickness/color to indicate confidence.

### Search & Knowledge Retrieval

**6. Wire up vector/semantic search via embeddings**
- **What:** Connect the orphaned `app/embeddings/` code to the search endpoint. On entity/claim upsert, generate embeddings via the configured embedding provider. Add a `/api/v1/search/semantic` endpoint that does cosine similarity search. Fallback to full-text when no embedding is available.
- **Why:** FTS + trigram search can't find "supply chain attack" when the entity is "CVE-2024-3094 XZ Utils backdoor". Semantic search bridges this gap. The embedding code is already written — it just needs connecting.
- **How:** Add a post-insert hook in the worker that calls `embedding_generator.embed()`. Store vectors in a `pgvector` extension column. The search endpoint does `ORDER BY embedding <=> :query_embedding`. Hybrid re-ranking: 0.6 × semantic + 0.4 × FTS score.

**7. Add CVE-to-entity graph linking for corpus documents**
- **What:** When a corpus document is ingested (CVE, Exploit-DB), extract CPEs, CWEs, and references from the JSON and create `references_cwe`, `affects_product`, `exploited_by` relationships to existing entities. Add a `corpus_document_id` FK on claims for traceability.
- **Why:** 396K corpus documents are disconnected text blobs. CVE-2024-3094 should be a first-class graph node linked to the `xz` product, the `supply_chain_compromise` technique, and the relevant actor. Without these edges, the corpus is a separate silo from the knowledge graph.
- **How:** Extend the corpus refresh worker to call extractors on `raw_json`. For each extracted entity, upsert and create a relationship. The existing extractors already handle CPE, CWE, technique IDs.

**8. Materialized view for search performance**
- **What:** Create a `mv_entity_search` materialized view that pre-joins entities with their latest claim description, tags, confidence, and CVSS. Refresh concurrently on a 5-minute cron. The search endpoint queries the MV instead of running 4 correlated subqueries.
- **Why:** The current search SQL runs 3 correlated subqueries per entity row against the claims table. With 44K entities × 118K claims, this is ~200ms per search. An MV cuts this to ~5ms.
- **How:** `CREATE MATERIALIZED VIEW mv_entity_search AS SELECT e.id, e.canonical_name, e.kind, ... FROM entities e LEFT JOIN LATERAL (SELECT ... FROM claims WHERE entity_id = e.id ORDER BY created_at DESC LIMIT 1) c ON true`. `CREATE UNIQUE INDEX ON mv_entity_search (id)`. `REFRESH MATERIALIZED VIEW CONCURRENTLY`.

**9. Add OSV fallback for CVE data freshness**
- **What:** In the `refresh_corpora` worker, when NVD returns 403/429, fall back to the OSV API (`https://api.osv.dev/v1/query`) or direct CVE.org JSON. Store the source in `corpus_documents.source_path` for traceability.
- **Why:** NVD's API is heavily rate-limited and unreliable. OSV aggregates from multiple sources (GitHub, PyPI, npm) and has no rate limit. Single-source CVE data creates blind spots.
- **How:** Add an `osv` provider to the corpus refresh worker. Try NVD first; on failure, query OSV. Mark the source in `raw_json._source`.

**10. RAG-backed `/ask` endpoint**
- **What:** Before sending the question to the LLM, retrieve relevant context from the knowledge base: search entities, claims, and corpus documents using the existing search pipeline. Inject the top 10 results as context. Add citation links in the response.
- **Why:** The current `/ask` is a naked LLM call with zero KB context. It can hallucinate threat intel. RAG grounds the answer in the platform's actual data.
- **How:** `context = await full_text_search(db, tenant_id, req.question, limit=10)`. Format as numbered sources. Prepend to the prompt: "Answer based on these sources. Cite by number." Return citations alongside the answer.

### Enrichment & Data Pipeline

**11. Add enrichment quality feedback and flagging**
- **What:** Add `quality_rating SMALLINT` (1-5 stars, NULL = unrated) and `flag_reason VARCHAR(200)` to `enrichment_cache`. Add `POST /api/v1/enrichment/{id}/rate` and `POST /api/v1/enrichment/{id}/flag` endpoints. Flagged entries are excluded from search results and graph context.
- **Why:** Enrichment data quality varies. Shodan may return stale port data. VirusTotal may flag a benign file. Without feedback, bad data persists for the full TTL and misleads analysts.
- **How:** Migration adds columns. The enrichment results API includes `quality_rating` and `is_flagged`. The lookup endpoint filters out flagged results.

**12. Auto-suggest lifecycle transitions from enrichment diffs**
- **What:** When `enrichment_diff.has_changes = true` for an IP or domain, and the diff shows the entity was previously flagged malicious but is now clean (or vice versa), auto-create a suggestion to transition the entity's lifecycle state. Surface as a notification, not an auto-action.
- **Why:** Enrichment data changes are the best signal for IOC decay. If GreyNoise says an IP is no longer in their noise dataset, that's a strong signal it's been re-allocated.
- **How:** Add a post-diff hook in the enrichment service. Compare `old_normalized.tags` vs `new_normalized.tags`. If "malicious" → "benign", suggest `retired`. Surface in the UI as a banner on the entity detail page.

**13. Sigma/YARA detection rule ingestion and TTP linking**
- **What:** Ingest Sigma rules from the SigmaHQ repository. For each rule, extract the `tags` field (which maps to MITRE ATT&CK technique IDs). Create `detects` relationships from the detection entity to the technique entity. Same for YARA rules with `rule_name → hashes` relationships.
- **Why:** The `detects`/`detectable_by` relationship types exist but are under-populated (3,826 edges for 1,400 techniques). SigmaHQ has 3,000+ rules with MITRE tags. This would dramatically increase detection coverage in the knowledge graph.
- **How:** Add a `scripts/import_sigma.py` that clones the SigmaHQ repo, parses YAML, extracts technique IDs, upserts detection entities, creates `detects` relationships. Wire into `make enrich-sigma`.

**14. Add adversary profile aggregation endpoint**
- **What:** Add `GET /api/v1/entities/{id}/profile` that, for `threat_actor` entities, returns a structured profile: all TTPs used, all malware deployed, all victims targeted, all campaigns conducted, with confidence levels and temporal scoping. Dedicated UI template `actor_profile.html`.
- **Why:** There are 920 threat_actor entities but no structured way to see "what does APT28 do?" The entity detail page shows raw JSON `external_refs`. Analysts need a human-readable adversary summary.
- **How:** Query relationships where `from_entity_id = :actor_id` and `kind IN ('uses', 'references_technique', 'deploys', 'targets')`. Group by relationship kind. Render in a tabbed UI (TTPs, Malware, Victims, Campaigns).

**15. Onion data classification and retention policy**
- **What:** Add `data_classification VARCHAR(20) DEFAULT 'unclassified'` to `parsed_documents` (values: `unclassified`, `sensitive`, `restricted`). Add automated CSAM hash check against Project Arachnid or PhotoDNA (API call, flag on match). Add `retention_days INT DEFAULT 90` and a weekly cron that purges expired Tor-sourced documents.
- **Why:** Tor scrape data is high-risk. It may contain CSAM, classified material, or PII. Without classification, retention limits, or automated detection, this is a legal liability.
- **How:** Migration adds columns. The Tor scraper sets `data_classification = 'sensitive'` by default. A weekly worker deletes documents older than `retention_days` for `source_type = 'tor_scrape'`. CSAM hash check runs as a post-scrape hook.

### API & Security

**16. Tighten CORS to known origins**
- **What:** Replace `allow_origins=["*"]` with `allow_origins=["https://z.je", "http://localhost:8000"]` (or read from config). Add `CORS_ORIGINS` to `.env` and `Settings`.
- **Why:** Wildcard CORS on a multi-tenant auth platform allows any origin to make authenticated requests if a user visits a malicious site. This is a CSRF vector.
- **How:** One-line config change. Add to `app/config.py`: `CORS_ORIGINS: str = "https://z.je"`. Parse in `main.py`.

**17. Per-API-key rate limiting**
- **What:** Add a `rate_limit` column to the `api_keys` table (default: 100 req/min). Implement a Redis-based sliding window rate limiter middleware. On limit breach, return 429 with `Retry-After` header.
- **Why:** No rate limiting means a compromised API key can drain enrichment budgets (VirusTotal: 500/day, Shodan: 100/day). One bad key = $0 of budget left for the day.
- **How:** Add `slowapi` or custom Redis middleware. Key = `rate:{api_key_hash}:{minute_window}`. Increment on each request. Return 429 when limit exceeded.

**18. Move path finding to recursive SQL CTE**
- **What:** Replace the `shortest_path` endpoint's "load all relationships → NetworkX" approach with a PostgreSQL recursive CTE: `WITH RECURSIVE path AS (SELECT ... UNION ALL SELECT ... FROM relationships JOIN path ON ...) SELECT * FROM path WHERE entity_id = :target LIMIT 1`.
- **Why:** Loading 124K rows into Python for every path query is O(n) memory. A SQL CTE runs in the database's optimized query planner with indexes. Expected speedup: 10-100×.
- **How:** Replace the `find_path` endpoint's implementation. Keep NetworkX as a fallback for complex path queries (weighted, multi-criteria). Add `WITH (path)` SQL. Limit recursion depth to 6 hops.

**19. Add graph summary stats endpoint**
- **What:** Add `GET /api/v1/graph/stats` that returns: total nodes, total edges, edge type distribution, top-10 most-connected entities, cluster count, average degree. Cache for 5 minutes.
- **Why:** Before loading a 124K-edge graph, analysts need to know what they're looking at. Summary stats let them decide whether to load the full graph or focus on a subgraph.
- **How:** Simple SQL queries with Redis caching. Return as JSON.

**20. Paginate and cap graph endpoints**
- **What:** Add `cursor` pagination to the graph endpoint. When no `entity_id` is specified, return only the most recent 500 relationships with a `next_cursor` for progressive loading. Cap focused BFS at `depth=3` by default.
- **Why:** Loading 124K edges into a browser will crash it. Even with the 900-node / 3000-edge caps, a BFS from a well-connected entity can still pull thousands of nodes. Progressive loading is the only way to make the graph UI usable at this scale.
- **How:** Change `build_graph` to accept a `cursor` (last relationship `created_at`). Use keyset pagination. The UI loads in batches of 500.

### Knowledge Graph Modeling

**21. Add campaign phase ordering**
- **What:** Add `phase_order INT` to relationships. For campaign-anchored relationships, set phase_order based on the kill chain stage (1=recon, 2=initial_access, 3=execution, ...). Add a `GET /api/v1/entities/{campaign_id}/timeline` endpoint that returns ordered phases.
- **Why:** Campaigns are currently flat collections of edges. An analyst can't reconstruct "first they phished, then they dropped Cobalt Strike, then they exfiltrated." The kill chain is the mental model for CTI; the graph should support it.
- **How:** Migration adds `phase_order`. The MITRE ATT&CK tactic ordering provides the canonical phase sequence. The timeline endpoint sorts by `phase_order`. The UI renders as a horizontal timeline.

**22. Schema-ify key `external_refs` fields**
- **What:** For the top-5 entity kinds by volume (domain, yara_rule, hash, malware, cve), extract frequently-queried `external_refs` fields into dedicated columns with proper types. E.g., for CVEs: `cvss_v3_score FLOAT`, `epss_score FLOAT`, `kev BOOLEAN`. Keep `external_refs` as a catch-all for everything else.
- **Why:** JSONB is flexible but unqueryable. "Show me all critical CVEs" requires scanning every entity's JSONB. Dedicated columns + indexes make this instant.
- **How:** Migration adds columns. Backfill from `external_refs`. Add partial indexes. The entity detail template reads from columns first, falls back to `external_refs`.

**23. Add `references_corpus` relationship type**
- **What:** When corpus documents mention entities (extracted during ingest), create a `references_corpus` relationship from the entity to the corpus document. This bridges the 396K-document silo to the graph.
- **Why:** Currently, the only link between corpus documents and entities is the `observed_in_document` claim, which points from entity → parsed_document (a different table). There's no path from the graph to the corpus.
- **How:** In the ingest worker, after entity extraction, create `references_corpus` relationships. Add a `corpus_references` section to the entity detail API response.

**24. Add `co_occurs_with` confidence decay**
- **What:** For `co_occurs_with` relationships (144 exist), implement time-based confidence decay: `effective_confidence = base_confidence × 0.95^months_since_last_seen`. Recompute on access, not on cron.
- **Why:** Co-occurrence is a statistical inference, not an attestation. Two IPs appearing in the same spam campaign in 2022 doesn't mean they're related now. Without decay, the graph accumulates stale inferences.
- **How:** Add a `last_observed_at` column to relationships. The graph visualization uses `effective_confidence` for edge opacity. Below a threshold (0.2), edges are hidden by default.

### UI & Analyst Experience

**25. Add a "where did I see this?" provenance view**
- **What:** For any entity, add a `GET /api/v1/entities/{id}/provenance` endpoint that returns: which documents mention it, which feeds it came from, which enrichment providers contributed, which analyst created it, when it was first and last seen. Dedicated UI tab.
- **Why:** The `observed_in_document` claims are low-value as assertions but high-value as provenance. "When did we first see this IP?" is the most common analyst question after "what is this IP?"
- **How:** Query claims with `entity_id = :id AND claim_type = 'observed_in_document'`. Join to `parsed_documents` for titles and dates. Query `audit_events` for who created the entity. Query `enrichment_cache` for provider history.

**26. Add a CVE detail page with NVD + EPSS + KEV aggregation**
- **What:** Enhance the existing `/cve/{cve_id}` template to pull from `corpus_documents.raw_json` (NVD data), the KEV list (is it in CISA's known exploited?), EPSS score, and any linked entities (affected products, exploiting actors). Render as a structured card, not raw JSON.
- **Why:** The CVE detail page currently shows raw data. Analysts need at-a-glance: CVSS score, EPSS probability, KEV status, affected CPEs, related threat actors. This is the most-visited page in any CTI platform.
- **How:** Extend the CVE detail API to aggregate from corpus_documents, enrichment_cache (NVD, KEV), and relationships. The template renders sections: Summary, Severity, EPSS, KEV, Affected Products, Related Actors, References.

**27. Add a breach/leak summary dashboard**
- **What:** Enhance the existing `breach_summary.html` to aggregate all Tor-sourced claims for a given victim. Show: what was leaked, when it was first seen, which ransomware group, what IOCs were extracted, link to the onion page (if still active). Add a `GET /api/v1/breaches` endpoint that lists all victims from Tor scrape data.
- **Why:** The breach summary template exists but is claim-scoped, not victim-scoped. Analysts want "show me all LockBit victims" not "show me claim abc-123."
- **How:** Aggregate claims with `claim_type = 'tor_site_findings'` by victim name (extracted from `value.deterministic.victims`). Group by threat actor. Add filters (actor, date range, sector).

**28. Add keyboard-driven search shortcuts**
- **What:** Add global keyboard shortcut `/` to focus the search bar. Add entity-type prefixes: `cve:CVE-2024-3094`, `ip:8.8.8.8`, `actor:APT28`. Add tab-autocomplete from recent searches.
- **Why:** Analysts live in the search bar. Every click away from the keyboard is friction. Prefix matching reduces result noise (searching "APT28" currently returns 20 mixed results; `actor:APT28` returns 1).
- **How:** JavaScript keydown handler for `/`. Parse prefix in the search endpoint. Add `kind` filter support to the search API (it already accepts `corpus`, just needs `kind` too).

### Observability & Operations

**29. Add data freshness monitoring dashboard**
- **What:** Add `GET /api/v1/health/freshness` that returns: last successful NVD sync, last KEV sync, last MITRE cache update, last Tor scrape, enrichment cache coverage (what % of entities have at least one enrichment), corpus document age distribution. Surface in the admin UI as a "Data Health" card.
- **Why:** The system has 37K completed ingestion jobs and 2.7K errors, but there's no dashboard to see "is my data current?" An analyst querying a CVE that was never synced is working blind. Data freshness is the foundation of trust in a CTI platform.
- **How:** Query `audit_events` for last successful sync per source type. Query `enrichment_cache` for coverage stats. Query `corpus_documents` for age histogram (SELECT date_trunc('week', published_at), count(*) GROUP BY 1). Cache results for 60 seconds in Redis.

**30. Add DB query performance monitoring and slow-query alerts**
- **What:** Enable `pg_stat_statements` in PostgreSQL. Add a `GET /api/v1/admin/slow-queries` endpoint that returns the top-20 slow queries from `pg_stat_statements`. Add an ARQ cron that checks for queries averaging >500ms and posts a Slack webhook alert.
- **Why:** The search SQL with 4 correlated subqueries, the graph BFS loading all relationships, and the path-finding NetworkX approach are all slow — but there's no observability into which queries are hurting most. Without measurement, optimization is guesswork.
- **How:** `CREATE EXTENSION IF NOT EXISTS pg_stat_statements` in a migration. The admin endpoint queries `pg_stat_statements` ordered by `mean_exec_time DESC`. The ARQ cron runs every 10 minutes and alerts on regressions.
